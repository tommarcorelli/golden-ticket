📖 GLOSSAIRE AD

  • Domaine / Domain Controller

    Le domaine est l'espace centralisé de gestion d'une entreprise (comptes, machines, droits). Le Domain Controller (DC) est le serveur qui fait autorité dessus.

  • Kerberos

    Protocole d'authentification d'AD : au lieu d'envoyer un mot de passe à chaque service, l'utilisateur obtient des tickets temporaires qui prouvent son identité.

  • SPN (Service Principal Name)

    Identifiant qui relie un service (ex: une base SQL) à un compte du domaine. Tout compte avec un SPN peut se voir demander un ticket par n'importe quel utilisateur authentifié.

  • Kerberoasting

    Attaque consistant à demander le ticket chiffré d'un compte de service, puis à le casser hors-ligne pour retrouver le mot de passe en clair — sans jamais alerter le système.

  • Domain Admin

    Groupe aux pleins pouvoirs sur le domaine entier. C'est la cible ultime de la plupart des attaques AD — la "couronne".

  • ACL (Access Control List)

    Liste qui définit qui a le droit de faire quoi sur un objet (lire, modifier, supprimer). Une ACL mal configurée peut donner à un compte anodin le droit de modifier un compte puissant.

  • Pass-the-Hash

    Technique consistant à réutiliser le hash d'un mot de passe (au lieu du mot de passe en clair) pour s'authentifier — le hash suffit, pas besoin de le casser.

  • krbtgt

    Compte spécial dont la clé chiffre tous les tickets Kerberos du domaine. Le compromettre permet de forger n'importe quel ticket à volonté : le vrai Golden Ticket.

  • Entra ID (ex-Azure AD)

    L'annuaire cloud de Microsoft. Même logique qu'un annuaire Active Directory sur site, mais hébergé par Microsoft et organisé en tenant (ex : corp.onmicrosoft.com).

  • App Registration / Service Principal

    Une application a elle aussi une identité dans Entra ID : l'App Registration définit l'application, le Service Principal est l'instance qui s'authentifie réellement, avec un clientId et un secret.

  • Application Administrator

    Rôle d'annuaire permettant de gérer les applications du tenant, y compris leurs identifiants. Un titulaire de ce rôle peut ajouter un secret à une application plus privilégiée que lui — et en emprunter les droits.

  • Global Administrator

    Rôle aux pleins pouvoirs sur un tenant Entra ID — l'équivalent cloud de Domain Admin. Ne devrait jamais être attribué directement à un compte d'automatisation.

  • OPSEC / SOC

    L'OPSEC (sécurité opérationnelle) désigne la discrétion d'un attaquant. Le SOC (Security Operations Center) est l'équipe qui surveille les journaux et peut détecter une intrusion — certaines actions (authentifications, changements d'annuaire) sont bien plus visibles que d'autres (cassage hors-ligne, par exemple).

  • AD CS (Active Directory Certificate Services)

    Service qui permet à une entreprise de faire tourner sa propre autorité de certification (CA) interne, notamment pour authentifier des comptes par certificat plutôt que par mot de passe.

  • Modèle de certificat (template)

    Définit qui peut demander (s'enrôler) un certificat auprès de la CA, et quelles informations d'identité ce certificat contiendra. Un modèle mal configuré peut accorder bien plus de pouvoir que prévu.

  • ESC1 (ENROLLEE_SUPPLIES_SUBJECT)

    Faille classique d'AD CS : un modèle laisse le demandeur choisir librement l'identité inscrite dans son certificat. S'il est en plus ouvert à tous, n'importe qui peut obtenir un certificat valide pour l'administrateur du domaine.

  • PKINIT

    Extension de Kerberos qui permet de s'authentifier avec un certificat plutôt qu'un mot de passe. Un certificat forgé pour une identité suffit à obtenir un ticket (TGT) valide pour cette identité.

  • Shadow Credentials (msDS-KeyCredentialLink)

    Attribut d'un compte qui stocke les clés d'authentification sans mot de passe (Windows Hello for Business, FIDO2). Un droit d'écriture sur ce seul attribut permet d'y injecter sa propre clé et de s'authentifier comme le compte, sans jamais toucher à son mot de passe.

  • Réplication / DCSync

    Les contrôleurs de domaine se synchronisent en répliquant leur base, secrets compris. Un compte détenant les droits DS-Replication-Get-Changes-All peut se faire passer pour un DC et répliquer le hash de n'importe quel compte — c'est l'attaque DCSync, possible sans être Domain Admin.
